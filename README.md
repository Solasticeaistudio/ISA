# ISA - Inogen Support Assistant

ISA is a Flask-based retrieval-augmented generation (RAG) support copilot for Inogen product documentation. It lets a user place product PDFs into a local `knowledge/` folder, run an ingestion script, and then ask documentation questions through a polished dark-mode chat interface. Answers are generated by Google Gemini and grounded in retrieved Pinecone chunks, with citations shown as clickable source chips.

The project was built as an interview-ready MVP: the modules are intentionally small, the retrieval pipeline is easy to trace, and the safety behavior is explicit. The public repository does not include private API keys, runtime logs, conversations, or proprietary PDF manuals. Users bring their own documentation and credentials.

## What ISA Does

ISA answers questions about Inogen product documentation. Instead of relying on the model's general knowledge, it retrieves relevant passages from uploaded PDFs and asks Gemini to answer from those passages only.

Typical questions:

- How do I replace the columns?
- Why won't my battery charge?
- What does this alarm mean?
- Compare the Rove 6 and G5.
- Is this device FAA approved?

ISA is intended for documentation support, not medical decision-making. It explains product documentation, troubleshooting steps, warnings, and procedures only when that information appears in retrieved source context.

## Core Features

- Flask web app with JSON chat endpoints.
- Google GenAI SDK for Gemini generation.
- Google embedding model support through `client.models.embed_content`.
- Pinecone vector database for semantic search.
- PyMuPDF PDF parsing with page-level extraction.
- Recursive ingestion of every PDF under `knowledge/`.
- Chunking that preserves page, file, category, product, and document type metadata.
- Stable vector IDs generated from file path, page number, and chunk index.
- Optional product and category filters during retrieval.
- Grounded Gemini prompt that requires citations and refuses unsupported answers.
- Clickable source chips that open the cited local PDF page when available.
- File-backed JSONL conversation memory.
- Dark-mode support-console frontend with prompt cards, filters, source pearls, pinboard, and scratchpad.
- Public-repo-safe `.gitignore` and `.env.example`.

## Architecture

ISA is split into three main layers.

### 1. Ingestion Layer

The ingestion layer lives under `ingest/`.

It is responsible for:

- finding PDFs in `knowledge/`
- extracting page text with PyMuPDF
- normalizing metadata
- chunking page text into retrieval-sized sections
- embedding chunks with Gemini
- creating or connecting to a Pinecone index
- upserting vectors and metadata into Pinecone

Important modules:

- `ingest/parser.py` - file discovery and PDF page extraction
- `ingest/chunker.py` - page-aware chunking
- `ingest/embedder.py` - Gemini embedding client, batching, pacing, and retries
- `ingest/utils.py` - metadata helpers, product guessing, logging, vector IDs
- `ingest/ingest.py` - CLI entry point

### 2. Retrieval and Generation Layer

The RAG layer lives under `rag/` and is called from `app.py`.

It is responsible for:

- embedding the user question
- querying Pinecone with optional metadata filters
- formatting retrieved chunks into model context
- building a safety-aware prompt
- generating an answer with Gemini
- returning deduplicated citations
- saving conversation turns locally

Important modules:

- `rag/retriever.py` - query embedding, Pinecone search, filter construction, context formatting
- `rag/prompts.py` - system prompt and chat prompt builder
- `rag/citations.py` - citation normalization and deduplication
- `rag/memory.py` - simple JSONL conversation storage

### 3. Web UI Layer

The web layer is a Flask app plus static frontend assets.

It is responsible for:

- serving the main chat page
- exposing chat, health, prompt, and document endpoints
- rendering assistant/user messages
- sending questions to `/api/chat`
- displaying source chips and source pearls
- linking citations to local PDFs via `/api/document/<relative_path>#page=<page>`

Important files:

- `app.py`
- `templates/index.html`
- `static/css/style.css`
- `static/js/chat.js`
- `static/wallpaper/isa-background.png`

## End-to-End Data Flow

### Ingestion Flow

1. A user places PDFs under `knowledge/`.
2. The user runs:

   ```bash
   python ingest/ingest.py
   ```

3. `find_pdf_files()` recursively finds every PDF.
4. `extract_pdf_pages()` opens each PDF with PyMuPDF and extracts text page by page.
5. Empty pages are skipped. Malformed PDFs are logged and skipped instead of crashing the full run.
6. `chunk_pages()` splits each page into retrieval-friendly chunks.
7. Each chunk receives metadata:

   - `filename`
   - `relative_path`
   - `category`
   - `page_number`
   - `chunk_index`
   - `product`
   - `document_type`
   - `text`
   - `snippet`

8. `stable_vector_id()` creates deterministic IDs from relative path, page number, and chunk index.
9. `GeminiEmbedder` embeds chunks in batches.
10. The ingestion script creates the Pinecone index if it does not exist.
11. Embedded vectors are upserted into Pinecone with metadata.
12. The CLI prints a summary of PDFs found, pages extracted, chunks created, vectors uploaded, and failures.

### Chat Flow

1. The frontend sends a POST request to `/api/chat`.
2. Flask validates the message and required environment variables.
3. `retrieve_context()` embeds the question as a retrieval query.
4. Pinecone is searched for the top matching chunks.
5. Optional product/category filters are applied if the UI selectors are set.
6. `format_context()` builds labeled source blocks for Gemini.
7. `build_chat_prompt()` combines the system prompt, retrieved context, and user question.
8. Gemini generates an answer from the retrieved context.
9. `build_citations()` returns citation metadata and snippets for the matched chunks.
10. The frontend renders the answer, citation chips, and source pearls.
11. Conversation turns are appended to `conversations/<conversation_id>.jsonl`.

Optional tuning variables used by the code:

```env
EMBEDDING_BATCH_SIZE=16
EMBEDDING_MAX_ITEMS_PER_MINUTE=80
EMBEDDING_MIN_BATCH_INTERVAL_SECONDS=0
EMBEDDING_QUOTA_COOLDOWN_SECONDS=65

CHUNK_OVERLAP_CHARS=250
```

Notes:

- `EMBEDDING_DIMENSION` must match the Pinecone index dimension.
- `text-embedding-004` commonly uses `768` dimensions in this project configuration.
- If you switch embedding models, verify the correct dimensionality and recreate or change the Pinecone index if needed.
- `PINECONE_NAMESPACE` is optional. Leave it blank for the default namespace.

## Adding Product PDFs

Put PDF files under `knowledge/`. The ingestion script scans recursively, so nested directories are allowed.

Suggested folders:

```text
knowledge/manuals/
knowledge/accessories/
knowledge/procedures/
knowledge/getting_started/
knowledge/faa/
knowledge/troubleshooting/
knowledge/warranty/
```

Category metadata comes from the first folder under `knowledge/`. For example:

```text
knowledge/procedures/96-09240-00-01_D_G5_Rove_6_Column_Replacement_Procedure.pdf
```

will produce:

```text
category = procedures
relative_path = procedures/96-09240-00-01_D_G5_Rove_6_Column_Replacement_Procedure.pdf
```

Category aliases are normalized. For example, `getting started`, `getting-started`, and `getting_started` map to `getting_started`.

## Running Ingestion

From the project root:

```bash
python ingest/ingest.py
```

The ingestion run prints a summary like:

```text
ISA ingestion complete
PDFs found: 17
Pages extracted: ...
Chunks created: ...
Vectors uploaded: ...
Failures: ...
```

A nonzero failure count does not always mean the entire run failed. ISA is designed to continue when one PDF is malformed, scanned image-only, or blocked by extraction issues.

### What Ingestion Stores in Pinecone

Each vector has:

```json
{
  "id": "isa-<sha256-prefix>",
  "values": [0.01, 0.02, 0.03],
  "metadata": {
    "filename": "manual.pdf",
    "relative_path": "manuals/manual.pdf",
    "category": "manuals",
    "page_number": 12,
    "chunk_index": 34,
    "product": "Rove 6",
    "document_type": "manual",
    "text": "Full chunk text...",
    "snippet": "Short preview..."
  }
}
```

Product guessing is filename-based. The current helper recognizes:

- Rove 4
- Rove 6
- G4
- G5
- At Home
- Voxi 5
- One G5
- Inogen One

Document type is category-based, with additional filename handling for FAA, warranty, and troubleshooting documents.

### `GET /api/document/<relative_path>`

Serves a PDF from the local `knowledge/` folder.

The `#page=12` portion is handled by the browser PDF viewer. Chrome and Edge usually support this behavior.

## Citation Model

ISA has two citation layers:

1. Inline model citations, such as `[1]` or `[2]`, generated by Gemini from the formatted context.
2. Structured citation cards returned by the backend and rendered by the frontend.

The backend citations are built from Pinecone metadata, not from model-generated text. This helps avoid hallucinated source cards.

Citation deduplication uses:

```text
filename + page_number + chunk_index
```

Each citation card includes:

- document name
- page number
- category
- product guess
- document type
- snippet
- retrieval score
- relative path for local PDF linking

## Safety and Scope

ISA is intentionally scoped to product documentation support.

The system prompt requires ISA to:

- answer only from retrieved context
- say it does not have enough information when context is insufficient
- avoid inventing specifications, warnings, warranty terms, FAA rules, or procedures
- avoid medical diagnosis
- avoid oxygen therapy or patient treatment advice
- recommend Inogen support, emergency services, or a healthcare provider when appropriate
- include citations

This matters because oxygen concentrator documentation can be safety-sensitive. ISA should assist with documentation lookup and product troubleshooting, not clinical decision-making.

## Troubleshooting

### Missing environment variables

Symptom:

```text
ISA is missing required environment variables.
```

Fix:

- create `.env`
- set `GEMINI_API_KEY`
- set `PINECONE_API_KEY`
- set `PINECONE_INDEX_NAME`

### Pinecone dimension mismatch

Symptom:

```text
Vector dimension does not match the dimension of the index
```

Fix:

- confirm `EMBEDDING_DIMENSION`
- confirm your embedding model's output dimensions
- create a fresh Pinecone index with the matching dimension, or use a different index name

### No citations returned

Likely causes:

- ingestion has not been run
- Pinecone index is empty
- wrong Pinecone index name
- wrong namespace
- filters are too narrow
- PDFs have no extractable text

Fix:

- run `python ingest/ingest.py`
- check `/api/health`
- try `All Products` and `All Categories`

### PDF extraction returns empty pages

PyMuPDF extracts embedded text. If a PDF is scanned image-only, it may need OCR before ingestion.

### Gemini quota or rate limit errors

The embedder includes pacing and retry settings. If quota errors continue:

- reduce `EMBEDDING_BATCH_SIZE`
- reduce `EMBEDDING_MAX_ITEMS_PER_MINUTE`
- increase `EMBEDDING_QUOTA_COOLDOWN_SECONDS`
- retry after quota resets

### Source chip opens the PDF but not the exact page

The app appends `#page=<page_number>` to PDF links. Browser PDF viewers handle that anchor differently. Chrome and Edge usually honor it; some viewers ignore it.

### Flask starts on the wrong port

Set `PORT`:

Windows PowerShell:

```powershell
$env:PORT="5001"
python app.py
```

macOS/Linux:

```bash
PORT=5001 python app.py
```

## Design Decisions

### Why Flask?

Flask keeps the application small and interview-readable. The route layer is simple enough to trace in one file while still supporting production-shaped API boundaries.

### Why Pinecone?

Pinecone provides managed vector search, metadata filtering, and serverless indexes. This keeps the MVP focused on RAG behavior rather than local vector storage mechanics.

### Why preserve page metadata?

Support answers need provenance. Page-level metadata allows ISA to cite exactly where information came from and link back to the source document.

### Why deterministic vector IDs?

Stable IDs make repeated ingestion safer. The same file path, page number, and chunk index produce the same vector ID, which avoids uncontrolled duplicate growth when re-ingesting the same material.

### Why store text in metadata?

For an MVP, storing chunk text directly in Pinecone metadata simplifies retrieval and prompt assembly. In a larger system, chunk text could move to object storage or a relational database while Pinecone stores IDs and metadata.

### Why file-backed conversation memory?

JSONL files are easy to inspect, debug, and reset locally. A production deployment could replace this with Postgres, Firestore, Redis, or another managed store.

### Why source links route through Flask?

The frontend should not expose arbitrary filesystem paths. `/api/document/<relative_path>` validates that the requested file is inside `knowledge/` and is a PDF before serving it.

## Known Limitations

- No OCR pipeline for scanned PDFs.
- No authentication or role-based access control.
- No admin UI for uploading and ingesting PDFs.
- No background job queue for long ingestion runs.
- No automated Pinecone cleanup for PDFs removed from `knowledge/`.
- Product guessing is filename-based and may need refinement for messy filenames.
- Browser support for PDF page anchors varies.
- Conversation memory is local JSONL, not multi-user durable storage.
- The app is a local MVP and uses Flask's development server by default.
