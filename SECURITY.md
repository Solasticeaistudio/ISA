# ISA Security Summary

ISA is an MVP with baseline safeguards. It should be described as security-conscious, not as a fully hardened enterprise deployment.

## Current Safeguards

### Secrets

- API keys are read from `.env`.
- `.env` and `.env.*` are ignored by Git.
- `.env.example` contains placeholders only.

### Private Documents

- PDF manuals are ignored by Git.
- Users provide local PDFs under `knowledge/`.
- Source document links are served through a controlled Flask route rather than exposing arbitrary filesystem paths.

### Grounded Answers

- The system prompt instructs ISA to answer only from retrieved context.
- Unsupported questions receive an insufficient-information response.
- Medical diagnosis and oxygen-therapy advice are out of scope.
- Backend code builds citations from Pinecone metadata.

### Product Safety

- ISA asks for model details before answering model-specific questions when no model is selected or named.
- This avoids broad answers for alarms, batteries, columns, FAA questions, and other product-specific workflows.

### Source Links

`/api/document/<relative_path>` validates that the requested PDF:

- resolves inside `knowledge/`
- exists as a file
- has a `.pdf` suffix

Invalid requests return `404`.

### Frontend Rendering

The frontend escapes user text, model answers, citation metadata, and snippets before rendering them. It does not render assistant output as trusted HTML.

## Prompt Injection Posture

ISA has basic prompt-injection resilience through grounding and source separation, but not a dedicated prompt-injection defense pipeline yet.

Current protections:

- The assistant is instructed to follow retrieved context only as evidence.
- The assistant has no execution tools or write actions available to users.
- Citations are generated by backend code, not model text.
- UI output is escaped.

Remaining risk:

- A malicious PDF could contain indirect prompt-injection text that appears in retrieved chunks.
- The current system does not scan uploaded PDFs for injection instructions.
- The current system does not run a separate claim-verification pass.
- The current system does not have auth, rate limiting, CSRF protection, or production security headers built into the ISA app itself.

## Recommended Hardening Roadmap

1. Add ingestion-time prompt-injection scanning for suspicious source text.
2. Add query-time checks for attempts to reveal prompts, secrets, or bypass citation rules.
3. Wrap retrieved chunks in explicit untrusted-evidence boundaries inside the prompt.
4. Add answer verification to ensure claims are supported by cited snippets.
5. Add authentication for admin/upload functionality.
6. Add rate limiting and CSRF protection.
7. Add deployment security headers and production WSGI hosting.
8. Add privacy controls for retained conversation logs.

## Interview Framing

A good way to describe ISA's security posture:

> ISA has responsible MVP safeguards: no committed secrets, ignored private PDFs, grounded prompting, backend citations, safe PDF source links, frontend escaping, and model clarification. For production, I would add layered prompt-injection defense, auth, rate limiting, source governance, and claim verification.
